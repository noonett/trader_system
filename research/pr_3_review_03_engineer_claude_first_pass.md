## 1. 介质选型工程现实

### 1.1 markdown + git GUI 封装的"封装"是非平凡的工程项目【M】

调研把"GUI 封装"当作"消除 git CLI 门槛的一个修饰词"——但工程上它实际上是一个独立的产品决策，工作量远高于"接两个按钮"：

| 候选 GUI 封装 | 实际能力 | 隐性成本 |
|---|---|---|
| Obsidian + Obsidian Git plugin | 自动 commit + 自动 push；新手友好 | plugin 由社区单人维护（denolehov），一旦弃坑无替代；与 Obsidian Sync 互斥语义复杂【W】 |
| VS Code / Cursor 内置 git | 完整能力但暴露所有 git 概念 | 不是封装，是"把 git CLI 改成按钮"——Perez De Rosso 的核心论点（概念错配）一个都没解决【M】 |
| GitHub Desktop / Sourcetree / Fork | 桌面端独立 GUI | 仍要求用户理解 commit / branch / push / pull 概念，对小白只是"少敲一行命令"【M】 |
| 自建 thin wrapper（一键"今日记录"按钮） | 完全隐藏 git | **是一个独立产品**——需要处理冲突、网络失败、密钥管理、平台差异【U】 |

**意见 1（M）**：调研中"markdown + git（GUI 封装）"是一个被低估了至少 1 个数量级的工程承诺。Obsidian + Obsidian Git plugin 是目前最接近"封装好"的开源组合，但**plugin 的核心维护风险**（社区单人项目）调研没有提；如果押注在它上面，5 年后维护中断的概率不可忽略。

### 1.2 Obsidian Git plugin 的工程现实【W → M】

- 移动端 Android 支持有但不稳定（基于 isomorphic-git，常见冲突错误）；iOS 完全不支持【W；社区论坛多源一致】。
- 大 vault（>10000 文件 + 大 .git）下 plugin 性能显著退化【W；GitHub issue 多次】。
- 与 Obsidian Sync 同时使用语义不清——容易产生重复 commit / 同步循环【W】。

**意见 2（W）**：如果 Phase 2 Design 选 Obsidian + Git plugin 作为基线，**必须现在就接受"5 年维护风险"和"移动端 fallback 弱"两个工程约束**。这不是"细节"——是"是否单端起步"决策的输入。

### 1.3 SQLite 路径在调研中被严重低估【M】

调研把 SQLite 标为 B+ 等级（弱于 markdown+git 的 A-），主要扣分点是"审计需要 app 层显式设计"和"门槛上要 app 层"。但工程现实是：

- **SQLite 上加 audit trail 是一个 50-100 行触发器 + INSERT-only event log 的标准模式**，不是高深工程【M；多个开源项目示范，如 datasette、oban、event-sourcing 模式】。
- **markdown + git 的"审计"在用户没设置 GPG 签名 + 私有远程 + 保护分支时实际等同于"承诺自律"**。调研附录提到了这一点，但综合矩阵里没扣分——这是工程不一致。
- **结构化查询（"过去 6 个月按情绪分组的胜率"）是 σ 系统未来必然要做的事**，markdown 上做这件事必然要 ETL 到 SQLite 或类似——也就是说**SQLite 不是"替代品"而是"必须迟早出现的层"**。

**意见 3（M）**：调研给 Obsidian (A) > markdown+git (A-) > SQLite (B+) 的排序，**忽略了"σ 5 年后必然需要结构化查询"这一不可逆约束**。更工程友好的判断是"主存储 = markdown，只读派生层 = SQLite"——而这恰好是调研 §3.3.5 提到但综合判断里没体现的方案。SQLite 的等级被低估了 1 档。

### 1.4 git 作为审计层的"虚假强度"

**意见 4（M）**：调研 §2.1.2 诚实地标注了"git 不是自动审计——需要 GPG 签名 + 推送 + 保护分支才是强审计"。但 §一矩阵和 §3.2 表格里 markdown+git 的审计列依然标 ✓——**这是把"理论上可达到的强度"当作"实际部署的强度"**。

工程现实：让一个零售交易者（小白）正确配置：
1. 生成 GPG key
2. 上传 GPG key 到 GitHub/Gitea
3. 配置 `git config commit.gpgsign true`
4. 设置远程私有仓库
5. 在远程开 protected branch
6. 配置 `force-with-lease` 行为约束

——这是 6 步，每一步都有出错可能。**没有一个开源 Obsidian plugin 把这 6 步打包成一键流程**。在用户没做这些配置时，"git 审计"≈"markdown 上多了一个时间戳序列"——比裸 markdown 强一点，但远不到 §一矩阵里"✓"标的水平。

## 2. SaaS lock-in 风险评估

### 2.1 调研对 Notion / Airtable / Trello 的批评是准确的，但优先级没排清【M】

调研 §三3 把 vendor lock-in 称为"非对称风险"。这个判断是对的。但工程上需要一个更细的优先级：

| 风险类型 | 概率 | 损失级别 | 净期望 |
|---|---|---|---|
| 公司倒闭 | 极低（Notion/Airtable 当前财务健康）| 灾难（数据全失）| 中 |
| 产品政策变化（涨价、砍 feature）| 中-高（Evernote 已发生）| 中（迁移成本，不是数据全失）| 高 |
| 账户被封 | 极低 | 灾难 | 中 |
| AI 子处理器变更（Anthropic/OpenAI 数据用途）| 中 | 中（隐私破坏，不是数据全失）| 高 |
| 数据格式锁定（CSV 导出残缺）| **必然发生**（Airtable CSV 不含附件）| 中（手动重建工作量）| **最高** |

**意见 5（M）**：调研把"账号被封"放在显眼位置（举的 LowEndBox 例子），但**对 σ 系统的真实最高工程风险是"数据格式锁定"**——这是 100% 概率发生的事（你迁移那一天才发现）。账号被封是极低概率事件。**如果调研要给 Phase 2 Design 一条最重要的工程警告**，应该是"任何中间格式都必须可被独立工具读懂，不依赖原 SaaS 还活着"。

### 2.2 Notion 帐号被封 ≠ 数据无法找回（精确性问题）【⚠️ 已 fact-check】

调研 §三3 写"Notion 账户被封 = 数据无法找回（GDPR 不覆盖用户创作内容）"——这个表述不准确。准确事实是：

- Notion 官方文档说明"workspace 删除后 30 天内可联系 support 恢复"【M; Notion Help Center】
- 帐号被**封**（vs 主动删除）的具体政策**没有在 Notion 公开文档说明**——LowEndBox 报道是 W 级单一事件【W】
- GDPR 是否覆盖"用户创作内容"是法律解读问题——GDPR Art. 15 (right of access) 在很多解读中**确实**覆盖用户上传/创作的内容，欧洲 DPA 多次裁决支持用户访问；但封号场景下能否实际行使是另一回事【M; 法律解读不一致】
- 实际工程结论：**Notion 不公开"封号后能否申诉拿回数据"的政策**——这本身就是足够的工程否决信号，但调研把它写成确定事实有过度自信问题

**意见 6（W）**：调研对 Notion 的批评方向正确，但表述强度过强。修正后：**Notion 不向用户提供"账户被封后能强制取回数据"的合同性承诺**——这就够说明问题，不需要更强的断言。

### 2.3 调研漏掉的两个真实工程级 SaaS 风险

**意见 7（W）**：调研在 SaaS 风险段落漏掉了两个交易日志特别相关的：

(a) **API 速率限制下的"数据可读性"现实**——Notion API 默认 3 req/s，意味着如果 σ 系统运行 5 年后用户有 ~5000 条记录，**一次完整 sync 就要 28 分钟**。这不是"理论可移植"，这是"实际不可工作"。【M; Notion API docs】

(b) **AI 子处理器一旦变更**——Notion AI 后端从 Anthropic 切换到任何其他公司时，用户的"诚实日志"会经过新公司处理，**没有任何用户提示机制**。这对一个明确以"AI 协作明确读时机"为约束的系统是更严重的 chilling effect 风险。【M; Notion 子处理器条款】

## 3. 系统级浮窗工程实现

### 3.1 macOS / Windows 浮窗的真实工程现实

调研 notes/11 给出了 4 类介入形态 × 5 类工具的可达性矩阵。从工程角度补充几个不准确/未提及的点：

**意见 8（M）**：在 macOS 上做"系统级浮窗 = Cmd+Space 唤起的 σ 输入框"的几个工程现实：

| 实现路径 | 工程难度 | 可达性 | 风险 |
|---|---|---|---|
| Raycast Extension（TypeScript API） | 低 | 仅 Raycast 用户（非默认）| 依赖 Raycast 运行【W】 |
| Alfred Workflow（脚本）| 低 | 仅 Alfred 用户（付费）| 同上 |
| 独立 Electron app + global shortcut | 中 | 任何用户 | 启动速度慢（Electron 冷启动 ~1-3s）；与"hot state 拦截"目标冲突【M】 |
| 独立 Tauri / Swift native app | 中-高 | 任何用户 | 启动快但 macOS 需要 accessibility 权限授权流程【M】 |
| macOS Shortcuts + Quick Note | 极低 | 默认能力 | UI 不可控；不能强制字段【M】 |
| Windows PowerToys / 自建 WinForms | 中 | Windows 用户 | 跨平台一致性需独立维护两套【M】 |

**关键工程现实**：调研里"浮窗"被当作单一形态，但实际上**至少 6 种实现路径，每一种 trade-off 不同**。把"浮窗 = 高潜力"的判断落到 v0 时，必须先选定具体路径。**最便宜的是 Raycast Extension**（如果用户已是 Raycast 用户）；**最贵但通用的是 Tauri native app**——后者从零起估计 80-160 工时（含跨 Win/Mac 测试）。【W】

### 3.2 iOS / 移动端是真正的物理边界【✅ 已 fact-check】

调研 §收敛信号 2 + notes/11 §5.2 的核心论断"iOS 沙盒禁止系统级强制介入"是正确的——这是工程硬约束，不是产品偏好：

- iOS 没有 `SYSTEM_ALERT_WINDOW` 等价权限【S; Apple Developer Documentation】
- 唯一例外是 Screen Time API + ShieldConfigurationDataSource，但这是家长控制框架，不能作为通用拦截层【M; Apple Developer docs】
- Live Activity / Dynamic Island 是显示层，不是介入层
- 私有 API（如 `_setApplicationIsOpaque:NO`）会导致 App Store 拒审【M】

**意见 9（S）**：调研这个判断是准确的——iOS 上做"sigma 浮窗在交易 App 上方提醒"在 2026 年仍然不可行。Android 上技术可行（System Alert Window）但 Google Play 审核越来越严，且各厂商 ROM（华为/小米/OPPO/vivo）有自己的限制。**对 A 股小白用户群（大量华为/小米/vivo 设备），Android 浮窗也不是"开发完就能用"——还要处理 OEM 启动管理、电池优化、自启权限**——这是调研未提的真实部署成本。

### 3.3 浮窗的"启动速度"约束被低估【W → M】

调研讲浮窗最高潜力时，说"决策时刻可达性最高"。但工程上有个被忽略的事实：

- Raycast 唤起时间：~50-150ms【W；个人测量】
- 自建 Electron 唤起：~500-3000ms（取决于是否预启动）【W】
- 自建 Tauri 唤起：~50-200ms【W】

**hot state 拦截窗口在 notes/11 被定义为"5-30 秒"。一个启动要 2 秒的浮窗，等用户从 hot state 拉回的能力远低于一个 100ms 的浮窗**。这不是边角参数——是"浮窗有效性"的核心。

**意见 10（W）**：如果选浮窗作为入口，**Electron 路径基本上不可行**（启动慢 + bundle size 大 + 与轻量化矛盾）。要么走 Raycast Extension（如果用户接受 Raycast 依赖），要么走 Tauri/Swift——后者工程量大很多。调研没有提这个分支选择。

## 4. A 股客户端工程可达性

### 4.1 同花顺 / 通达信 / 东方财富 / 文华财经的工程现实

**意见 11（W → M）**：调研引用"同花顺 ~21M MAU"作为 A 股工程不可触达的论据【M】。这个数字 fact-check 过（详见 §10）准确。但**调研在工程方案上几乎完全没讨论 A 股客户端的可介入性**。我能给的实战观察（标 ❓ 因为我对最近版本不熟）：

| 客户端 | 平台 | 第三方插件接口 | 浏览器扩展可达性 | API 自动化 |
|---|---|---|---|---|
| **同花顺 PC（Windows）** | Win32 native + 自有脚本语言 | 同花顺**公式系统 / iFinD 数据接口**（付费）❓不熟悉具体能力 | ❌ 浏览器扩展不可达 | iFinD 有 Python SDK（付费）【W】 |
| **通达信 PC** | Win32 native | TDX 公式语言（封闭）| ❌ | 第三方爬数据是常见做法（pytdx）【W】 |
| **东方财富 PC** | Win32 native + 部分 Web 模块 | ❓ 不熟悉是否有官方插件 | △ 部分 Web 页面可达 | 第三方爬数据【W】 |
| **东方财富 Web (eastmoney.com)** | 浏览器 | ✅ Chrome Extension 可注入 | ✅ | API 反向工程【W】 |
| **文华财经（期货）** | Win32 native + 麦语言/M1 | 麦语言可写自定义指标和交易系统 | ❌ | 文华提供模型导出但不开放下单 API ❓ |
| **券商统一终端**（如华泰、招商、银河 PC 客户端）| Win32 native，多基于通达信内核 | ❓ 通常不开放 | ❌ | 部分券商提供 OpenAPI（如华泰矩阵、东方财富 OpenAPI 通道）❓ |

**关键工程结论**：

- **同花顺 PC 和通达信 PC 是 Win32 native，浏览器扩展完全不可达**——调研这个判断准确【M】
- **A 股下单端基本没有零售可用的 broker API**（与美股 Alpaca / IBKR API 不同）——任何"自动从交易记录注入到 σ 系统"的设想在 A 股都极其受限【M; 多源一致】
- **AKShare / Tushare 提供市场行情数据 API**【M; akshare.akfamily.xyz / tushare.org】，但**这是行情数据，不是用户的成交记录**——σ 系统需要的是后者。
- 用户成交记录的工程现实是：**手动 CSV 导出**（券商 App 通常每月导出一次）+ **OCR/截图 / 微信通知抓取**（极脆弱）【W】

### 4.2 调研在 A 股工程层完全空白的两个具体盲区

**意见 12（U → 重要）**：调研把"A 股 PC 客户端 浏览器扩展不可达"作为唯一的 A 股工程结论。但 v0 落地必须解决的两个具体工程问题，调研一个字没提：

(a) **σ 系统如何获得用户的真实成交记录**？
   - 选项 A: 用户手动 CSV 导入（多数券商每月可导出 Excel；门槛低但延迟大）
   - 选项 B: 用户手动 OCR 委托单（即时但极不可靠）
   - 选项 C: 某些券商有 OpenAPI（如华泰矩阵——专业版）❓不熟悉零售开放程度
   - 选项 D: 完全用户手动输入（最简单，但与 σ 系统的"客观字段必填"目标冲突——客观数据靠用户输入会漏）

   **v0 几乎肯定走选项 A 或 D**——但这不是"细节"，这决定了"σ 是当日记录系统还是周复盘系统"。

(b) **σ 系统如何在"决策前"和券商 App 共存**？
   - PC 同花顺/通达信用户：σ 必须独立窗口，物理 alt-tab；浮窗是唯一的"同屏"形态【M】
   - PC 东方财富 Web 用户：浏览器扩展可行【M】
   - 移动同花顺/雪球用户：iOS 完全不可达；Android 系统浮窗在国产 ROM 下需要复杂权限授权【W】
   - 期货用户（文华）：Win32 浮窗可行【M】

**意见 13（U）**：调研缺一个明确的"v0 不试图解决移动端"声明。从工程现实看，σ v0 应该明确**只覆盖 PC 桌面（Windows + macOS）+ 周/月异步复盘走任何介质**——不要试图给 iOS 同花顺用户做实时介入。试图覆盖移动端会让 v0 工作量翻 3-5 倍且大概率失败。

### 4.3 微信生态的工程黑洞【W】

**意见 14（W）**：调研明确说"冲动种子常在微信群"，但工程上微信生态对第三方介入是几乎完全封闭的：

- 微信 PC 客户端是 native，没有公开扩展接口【M】
- 微信小程序由腾讯审核（金融类小程序需要资质）【M】
- 企业微信有 API，但个人微信没有【M】
- 第三方"微信助手"类工具（如 wxBot）违反 ToS，账号有被封风险【W】

**含义**：调研中"冲动种子在微信群"的事实成立，但**没有任何工程方案能在微信群消息流上做拦截**——这意味着 σ 系统的"盘前规则书写"必须假设"用户已经看过群消息后才坐下来做规则"，而不是"σ 在群消息触发时介入"。这是 v0 必须接受的工程约束，调研没有点破。

## 5. 三个位置组合的工程总成本

调研提出"σ 入口 = 盘前规则书写 + 盘中拦截 + 盘后 EMA + 异步深度"四层组合。从工程角度算这个组合的真实成本：

### 5.1 每层的最小工程估算（个人 N=1 项目，单人开发）【W】

| 层 | 最简形态 | 工程量估算（自托管单人）|
|---|---|---|
| 盘前规则书写 | Markdown 模板 + 提醒（系统通知）+ git commit hook | 2-8 工时 |
| 盘中拦截 | 浏览器扩展（仅 TradingView Web / 东财 Web 用户）| 40-80 工时（含跨浏览器适配）|
| 盘中拦截 | Raycast Extension（macOS 用户）| 20-40 工时 |
| 盘中拦截 | Tauri 跨平台浮窗 | 80-160 工时（含跨平台测试）|
| 盘后 EMA | 同盘前规则书写工具，加 ≤4 字段 form | +4-8 工时 |
| 异步深度 | 现有 Obsidian / VS Code + git | 2-4 工时（配置 + 模板）|
| **总计（最简组合：盘前 + Raycast 浮窗 + 盘后 EMA + Obsidian 异步）** | | **30-60 工时** |
| **总计（要做"通用"跨平台 v0）** | | **130-260 工时** |

**意见 15（W）**：调研说"按时刻分层"——这个判断方向对，但工程总成本调研一字未提。从 N=1 单人开发的现实看：
- **如果只做 macOS + Raycast 用户子集**：30-60 工时是可达的（即"v0 可在 1-2 月业余完成"）
- **如果要做 Windows + macOS 通用版本**：130-260 工时（即"v0 至少 4-6 月业余"）
- **如果要做 PC + 移动 双端**：再翻 2-3 倍，且移动端有效性接近 0（见 §4）

**v0 工程理性选择是前者**——但这意味着用户群被严格限定在"macOS + Raycast 用户"。调研把"按时刻分层"当作普适设计原则，**没有评估它对工程成本和用户群范围的级联效应**。

### 5.2 多介入点的"互相干扰"问题【M】

**意见 16（M）**：调研讲"按时刻分层"时假设各层独立，但工程上多个介入点存在干扰：

- 盘前规则 + 盘中拦截 + 盘后 EMA + 异步复盘 = 用户每天接触系统 3-4 次
- engagement-effectiveness paradox（调研自己引用的 Linardon 2024）说高频接触不等于高效果【S】
- mHealth 文献中 "burden" 是 dropout 主因（Wrzus 2023, Honary 2023 已在调研中引用）【S】
- 调研 §收敛信号 1 自己说"σ 入口按时刻分层组合"——但**调研里没有任何文献验证"4 次/天接触的训练系统留存率"**

调研假设三层组合不会互相吃掉留存预算。这是一个**未验证的工程假设**，应该明确标注。Phase 2 Design 必须为"可能只有 2/4 层被实际使用"做 fallback 计划，**不能假设用户会同时坚持 4 层**。

## 6. AI 集成工程边界

### 6.1 "AI 周期性透明读"的工程实现【M】

调研多次说"AI 周读 + 透明告知边界 = 中等被审视感"。从工程上具体落地这个机制：

| 实现路径 | 工程现实 | 隐私 trade-off |
|---|---|---|
| 用户主动调用 ChatGPT / Claude，复制粘贴 | 0 工程量 | 全文走云 LLM；与"本地优先"矛盾 |
| 用 Claude API / OpenAI API + 本地脚本 weekly cron | 4-8 工时 | 同上，但更结构化 |
| Cursor / VS Code 内调用 LLM（用户已用 Cursor）| 0-2 工时 | 与 Cursor 后端隐私政策绑定 |
| 本地模型（Ollama + Llama 3.1 8B / Qwen 2.5 7B）| 8-20 工时 | 真正本地 |
| 自建 RAG（LlamaIndex 之类）+ 本地模型 | 40-80 工时 | 完全本地，但开销大 |

**意见 17（M）**：调研对 AI 集成的工程边界没有具体讨论。"AI 周读"听起来温和，但**默认实现路径几乎肯定是"调云端 LLM API"**——这与调研同一份文件中"本地优先 / 中等被审视感 / 不依赖第三方'我们承诺不读'"的隐私原则**直接冲突**。

要真正落地"中等被审视感"约束，**v0 应当默认走 Ollama + 7-8B 本地模型**。这意味着用户最低硬件要求（M1/M2 Mac / 32GB RAM Windows + GPU）——这条没在调研中讨论，但是工程现实。

### 6.2 "git history 中的修改痕迹检测"的工程现实【W】

调研 §3.3.6 说"由 AI 或脚本周期性检测：哪些'决策链'或'交易后情绪'文件在 24 小时后被修改？"

**意见 18（W）**：这个机制工程上是 trivial 的（`git log --diff-filter=M --since=...`），但**有效性的核心约束在调研里被忽略**：

- 用户可以 `git rebase -i` 静默修改提交时间，让"24 小时后修改"看起来像"当时就那么写的"【M; git docs】
- 用户可以 `git commit --amend` 修改 commit 内容【M】
- 只有当推送到**有保护分支的远程**且本地不能 force-push 时，这条防御才工作【M】
- 没有 GPG 签名时，commit 作者可被任意伪造【M】

也就是说调研提到的"git 是 σ 系统能力"必须依赖前面 §1.4 提到的 6 步配置都完成。对一个**故意想骗自己**的用户，这些配置全部由用户自己掌控——他可以选择不签名、不推送、不开保护分支，然后静默 rebase 自己的过去。**git 在"防止用户骗自己"这件事上的工程强度比调研暗示的弱很多**。

## 7. PGSI / PHQ-4 工程实现

调研 §四 / 收敛信号 3 + notes/12 强烈要求："临床高风险用户必须 default No-AI"——基于 PGSI 中-高、PHQ-4 高等阈值。从工程角度：

### 7.1 PGSI / PHQ-4 落地的工程现实

**意见 19（M → 重要）**：

- PGSI（Problem Gambling Severity Index）9 题；PHQ-4（Patient Health Questionnaire-4）4 题——都是公开量表【S】
- 工程实现：4-9 个 form fields + 加分逻辑——**工程量 < 4 工时**【W】
- 但**关键工程问题没在调研中讨论**：
  - 谁来执行筛查？σ 系统自动化执行？还是要求用户手动找心理医生？
  - 筛查是入门一次还是定期重测？v5 §三.11 提到"3 月稳定+主动书面知情同意三道关"——意味着至少要持续追踪
  - 筛查结果如何存储？这是高度敏感的健康数据，比交易日志敏感得多
  - 如果筛查显示中-高风险，"自动回退到 No-AI"的具体技术机制是什么？关闭某个 feature flag？需要工程上是不可被用户绕过的吗？

**意见 20（M）**：调研把 PGSI/PHQ-4 当成"已落地的设计原则"——但工程上它**还有一整层未解决的问题**（谁评估、何时重测、数据存储敏感度、不可绕过性）。Phase 2 Design 必须明确这些。**最简实现是"用户自评 + 自报，不存储原始答案，只存最终是否触发 No-AI 标志"**——但这意味着用户可以"装健康"绕过 No-AI 强制，与"临床安全"目标矛盾。这是真实的工程困境，调研没暴露。

### 7.2 量表本身的工程注意点

- PGSI 是 12 个月回顾问题，不是即时状态——意味着**不能用作"今天是否高风险"的实时检测器**【S; PGSI 设计文档】
- PHQ-4 是 2 周回顾——同上【S】
- σ 系统真正想做的"实时检测高风险"需要更短窗口的量表（如 PHQ-2 当日版）❓ 我对短窗口版本的临床有效性不确定

**意见 21（U）**：调研用 PGSI/PHQ-4 作为"高风险用户分流"——这个工具选择对**入门筛查**合适，对**实时风险检测**不合适。两者工程实现差异很大，调研把它们混为一谈。

## 8. "git commit 仪式 + AI 周期透明读"工程现实

### 8.1 "commit 仪式"的工程实现意外复杂【W → M】

调研 §收敛信号 5 把"git commit 仪式"作为"中等被审视感"的核心机制。工程上"commit 是仪式"要真正起作用需要：

- commit message 不为空（git hook 可强制）【M】
- commit message 遵循模板（如"trade-decision YYYY-MM-DD-HHMM 标的"）（git hook 可强制）【M】
- commit 后强制推送（GUI 自动化或 hook）【W】
- commit 时**不能 amend / squash / rebase**（不能在 git 层强制——只能远程 protected branch + 不允许 force push）【M】
- commit 时显式确认"我承诺如实记录"——这是 honesty oath，需要 hook 拦截【W】

**意见 22（M）**：调研里"git commit 仪式"轻描淡写，但**所有这些约束加起来 ≈ 一个完整的小型 commit policy 系统**。开源工具有零碎组件（pre-commit、commitlint、commitizen）但没有打包好的"trader honesty commit policy"。从零搭起这套估计 10-25 工时【W】，调研里这个数字不存在。

### 8.2 "周复盘 AI 透明读"的工程实现【M】

- "周日 21:00 让 AI 读这一周的所有交易日志"——这是 cron + 脚本，工程量低【W】
- 但"读完后做什么"是核心问题：
  - 给用户结构化反馈（最少冲击）
  - 给用户写一份模板化总结（中等冲击）
  - 给用户提醒高风险模式（高冲击 + 谄媚 / wise feedback 风险）

调研 §三 已有约束（v5 §三.2 反馈不主动判断好坏；wise feedback 仅在月度等）——工程实现要让 AI 在 prompt 层"少说"是已知的难点（LLM 倾向 over-helpful）。

**意见 23（M）**：调研把"AI 周读"当作温和动作。但**让一个 LLM "读完后什么都不主动做" 是 prompt engineering 上反直觉的**——LLM 默认是 helpful。要让它产生"非谄媚 / 非过度建议"输出，需要明确的 system prompt 约束 + 输出 schema 限制 + 多次评测。**这个 prompt engineering 工作量在调研里完全是 0**——但实战上估 20-40 工时【W】。

### 8.3 "中等被审视感"在工程上是脆弱的

**意见 24（U）**：调研 §三5 收敛信号说"markdown + git commit 仪式 + 私有 + AI 周期性透明读 ≈ 中等被审视感"。从工程上看，这个"≈"号包含很多变数：

- 如果用户**关闭** AI 周读 → 退化到"完全私密"端 → Lelkes 风险（敷衍）
- 如果用户**让多个 AI 读** → 走向"模糊监视" → chilling effect
- 如果用户**不做 commit message 仪式** → 退化到"纯 markdown"

这意味着"中等被审视感"不是一个稳定均衡——它**依赖用户持续维持几个相互独立的工程行为**。任何一个掉链子都偏向不健康端。**工程实现必须把这几条做成"默认开启 + 关掉时给警告"，而不是"用户随意配置"**——这是调研未给出的具体工程约束。

## 9. Final Verdict

### 9.1 最简单可行的 v0 工程方案

**直接回答**：v0 应该是**"单端 + 单形态 + 单介质 + 完全无 AI"**。

具体技术栈（推荐）：
```
入口形态：Obsidian vault + 模板（盘前 / 盘后 EMA / 周复盘 / 月校准 4 个模板）
介质：本地 markdown，私有 GitHub repo（手动 push，不强制自动化）
GUI 封装：Obsidian 默认界面 + Obsidian Git plugin（自动 commit）
AI：完全无（v0 阶段）—— 周复盘靠用户自己读模板
盘中拦截：仅"早晨规则书写"作为必经之路；不做盘中浮窗
盘后 EMA：Obsidian 内同 vault 一个 daily note，≤4 字段
盘前规则：固定时间系统通知（macOS Calendar / Windows Task Scheduler）+ Obsidian 模板
PGSI/PHQ-4：入门一次性自评 form（Google Form / 静态 HTML），存于 vault
异步深度：周复盘 / 月复盘 走相同 vault，git commit 标记
```

**v0 工作量估算（单人业余）**：
- Obsidian 模板设计 + 测试：8-15 工时
- Obsidian Git 插件配置 + 文档：2-4 工时
- 盘前定时通知配置：2-4 工时
- PGSI/PHQ-4 自评 form：4-8 工时
- 操作手册（小白可上手）：8-15 工时
- N-of-1 调试 + 4 周自测：30-60 工时（被使用，不是开发）
- **总开发：~25-50 工时**
- **总占用（含自测）：~55-110 工时**

**v0 故意不做**的事：
- 盘中浮窗（高工程成本 + 移动端不可达 + Phase 2 Design 应再讨论）
- 自动 broker 集成（A 股零售无可用 API）
- AI 集成（Phase 1 v5 + 调研都说临床高风险用户 default No-AI；v0 一律 No-AI 最安全）
- 多端同步（单端起步证据更强；多端故障模式工程上昂贵）
- 精细审计（GPG 签名 + 保护分支等可后续加）

**v0 用户群限制**：
- macOS 或 Windows 桌面用户
- 接受手动写 markdown
- 接受每月手动从券商 App 导出 CSV（不做自动化）
- A 股 / 港股 / 期货任意——但盘中介入（v0 不做）只对 Web 端券商有效

**意见 25（W）**：v0 越简单越好。即使是上面这套已经包含 4 个层（盘前 / 盘后 EMA / 周 / 月）+ Obsidian + git——已经接近"过度复杂"的边缘。**v0 真正最简版甚至可以省掉 Obsidian Git plugin，让用户自己 commit**——只是对小白这个门槛太高。

### 9.2 调研中最被高估的工程方案

**直接回答**：**"系统级浮窗"是调研中最被高估的工程方案**。

理由：
1. 调研自己承认"作为行为日志入口的学术证据为 0"【调研 §一收敛信号已承认】
2. 工程实现路径分裂（Raycast / Alfred / Tauri / Electron 等 6+ 种），每种 trade-off 不同
3. 跨平台工程量大（Win + Mac 通用至少 80-160 工时）
4. 移动端不可达（iOS 沙盒 + Android OEM 限制）—— σ 用户群有相当比例在用手机交易
5. 启动速度约束被低估（Electron 路径基本不可行）
6. 即使做出来，调研中"决策时刻可达性最高"是基于**用户已经在桌面上**——但 A 股移动下单用户群完全不在桌面

调研对浮窗的"高潜力"判断在工程现实里**仅对一个用户子集（macOS + Raycast 用户 + Web 端券商）成立**——这个子集在 σ 用户群中可能 < 5%。

### 9.3 调研中最被低估的工程方案

**直接回答**：**"早晨规则书写 = 主入口"是调研中最被低估的工程方案**。

理由：
1. 调研 §收敛信号 6 + Patterson 2020 RCT + Masicampo 2011 都强力支持"用户预先承诺 + 必经之路"【S】
2. 早晨规则书写工程上**几乎零成本**——一个 markdown 模板 + 一个系统提醒
3. 不依赖任何介入形态（不需要浮窗、不需要扩展、不需要 broker API）
4. 跨平台天然兼容（Obsidian / VS Code / 任何文本编辑器）
5. 与 v5 §三.3 binding pre-commitment 直接对齐
6. 调研 §收敛信号 4 自己说"事后回忆系统性漏掉最危险的冲动交易"——意味着事前规则是结构上唯一可靠的拦截
7. 调研多次提"实施 intention 已证 d≈0.65"——这是 σ 系统能从 v0 立即拿到的最高 ROI 单一组件

调研虽然把"早晨规则书写"列为四层之一，但**没有把它和其他三层做"工程性价比"对比**。从工程性价比看：
- 早晨规则书写：0-5 工时 / d≈0.65 效应（如果 implementation intention 文献迁移成立）
- 盘中浮窗：80-160 工时 / 效应未知
- 盘后 EMA：4-8 工时 / 效应未知（在调研中"不填本身也是数据"——这是承认效应可能小）
- 异步深度：2-4 工时 / 效应中等（Pennebaker 系列）

**早晨规则书写的工程 ROI 比其他三层高 1-2 个数量级**——这意味着 v0 应该把所有工程预算压在它上面，其他三层做"最简版本 + 后续迭代"。调研把四层并列处理，模糊了这个 ROI 差异。

## 10. 通用 5 尖锐问题

### Q1. Phase 2 调研的"6 维度对比矩阵"是否有重大方法学瑕疵？

**意见 26（M）**：notes/07 §一矩阵的方法学瑕疵从工程视角看：

(a) **维度不正交**：认知深度、留存、错误率/数据质量、摩擦类型、交易场景适配——其中"交易场景适配"实际包含了"决策时刻可达性 + 工作流兼容 + 设备 affordance"三个独立维度，被压缩成一格。

(b) **缺关键工程维度**：跨平台一致性、维护成本、生态依赖度、长期数据可读性、安全/隐私实施现实——这 5 个维度对 σ 系统的 v0 决策都很重要，但矩阵里都没有。

(c) **单元格"高/中/低"不可比**：例如 CLI 的"持续摩擦：高"和 Mobile 的"持续摩擦：低"——这两个低/高的判断是基于不同样本（CLI 是开发者群，Mobile 是普通群）——不能直接比较。

但这条瑕疵 **不致命**——调研在 §一矩阵末尾承认"所有'+/-'都是相对大小，不是绝对断言"，这是合理的诚实标记。

### Q2. "入口按时刻分层组合"（盘前/盘中/盘后/异步）的合成结论是否过度自信？

**意见 27（U → 重要）**：从工程视角看，**有过度自信问题**：

(a) 没有任何文献验证 4 层组合的留存
(b) 调研自己引用的 engagement-effectiveness paradox（Linardon 2024）应当让"4 次/天接触"成为警告而非默认
(c) 工程上"用户每天接触 4 次"是高门槛习惯——5% 留存基线在 finance 类目（调研引用 AppsFlyer 2024）适用于**一次/天**接触
(d) 调研 §收敛信号 1 把 4 层并列处理，没有给"如果用户只能坚持 1 层，应该是哪 1 层"的工程性价比答案

**应改为**：4 层是 **ideal**，但 v0 应该明确"先做最高 ROI 的一层（早晨规则书写），其他 3 层是 v1+ 的逐步增量"。

### Q3. No-AI 必须作为高风险用户默认路径的论证是否足够？

**意见 28（M）**：从工程角度论证**充分**——v5 §三.11 + notes/12 + 跨调研收敛——但**工程实施路径不充分**（见 §7）。"No-AI 默认"在 v0 几乎自动满足（v0 本来就不集成 AI）。问题在 v1+ 引入 AI 时如何**强制**No-AI 模式不能被用户简单绕过——这部分调研没解。

### Q4. 本调研的 6 份子笔记由同一模型生成——是否产生了系统性偏见？

**意见 29（U → 同模型 self-aware）**：作为同模型 reviewer，我能识别但不能完全避免的同模型偏见：

(a) **对 git / markdown / Obsidian 的过度青睐**——这些是开发者文化中"高品味"工具，模型训练数据里被技术博客大量正面引用。但对 A 股小白零售交易者这个用户群，模型可能没有足够"普通中文用户实际工作流"训练数据来纠正这个偏好。

(b) **对"按时刻分层"这种结构化方案的过度青睐**——模型偏好分层 / 分类 / 矩阵化的解决方案，因为这种结构在训练文本里普遍。但简单的"做一件事，做透"在交易训练系统里可能比"4 件事都做"留存更好——这是模型可能漏掉的简单解。

(c) **对 SaaS 的过度敌视**——SaaS lock-in 风险确实存在，但模型训练数据里 SaaS 批评比 SaaS 优势在比例上偏多（开发者文化 bias）。对一个真正的小白，Notion 的"开箱即用"门槛低这一项，可能比模型评估的更重要。

(d) **对"工程 trade-off"的回避**——模型倾向给"两边都好"的方案（例如"markdown + git GUI 封装兼有低门槛和强审计"）——但这个组合在工程上不存在（GUI 封装的工作量是真正的成本）。

### Q5. 针对中国 A 股 + 期货用户群的本地化盲区是什么？

**意见 30（W → 关键盲区）**：作为同模型 reviewer 我**自己就有这个盲区**——以下是我能识别但不一定能完全填补的：

(a) **微信生态不可介入**——已在 §4.3 讨论
(b) **A 股下单端无 broker API**——与美股完全不同，调研没明确画这条边界

## 12. Fact-check 5 项关键事实

按用户要求，对 5 项核心工具/平台事实做 fact-check：

### Fact 1：同花顺 / 通达信 ~21M MAU

**调研引用**：notes/11 §2.1 "同花顺：~21M MAU；东方财富：~19M MAU；涨乐财富通：~11M MAU；大智慧：~6M MAU（2025 数据）"

**Fact-check 结果**：✅ **准确**（甚至略保守）

- Web 检索证实：2025 年监测数据显示同花顺 MAU 2107.9 万（约 21M），东方财富 1888.2 万（约 19M），大智慧 574.5 万（约 6M）【M; 月狐数据 / 网易报告 2025】
- 2025 年 9 月证券服务类 App 总月活 1.75 亿，同比增长 9.73%【M; 新浪财经 2025-10-21】
- "通达信"未单独检索 MAU，但作为 PC 桌面客户端，MAU 概念不直接适用——通达信内核被多家券商终端使用

**修正**：调研把"通达信"和"同花顺"并列说"~21M MAU"是准确的同花顺数字，通达信本身没有单独 MAU 数字（它是 PC 客户端 + 行业基础架构）。**精确表述应为"同花顺 ~21M MAU；通达信作为 PC 客户端是 A 股专业散户主流但无独立 MAU 公开"**。

### Fact 2：iOS 沙盒禁止系统级强制介入

**调研引用**：notes/11 §5.3 "iOS 沙盒禁止 system overlay（除非 PiP / Live Activity，但都不是介入用途）"；entry_form 收敛信号 2 + §四 4.2 多次

**Fact-check 结果**：✅ **准确**

- iOS 没有 Android `SYSTEM_ALERT_WINDOW` 等价权限【S; Apple Developer Documentation】
- 唯一例外是 Screen Time API + ShieldConfigurationDataSource——是家长控制框架，**不能作为通用拦截层**【M; Apple Developer docs】
- 私有 API 会导致 App Store 拒审【M; 多源开发者社区】
- PiP / Live Activity / Dynamic Island 是显示层非介入层【M】

**补充**：调研此点的工程结论"iOS 上做'σ 浮窗在交易 App 上方提醒'不可行"是 2026 年仍然成立的硬约束，不是产品偏好。

### Fact 3：Notion 账户被封 = 数据无法找回（GDPR 不覆盖用户创作内容）

**调研引用**：entry_form §一收敛 5 + notes/08 §三.3 "Notion 账户被封 = 数据无法找回（GDPR 不覆盖'用户创作内容'）"

**Fact-check 结果**：⚠️ **部分准确（强度过强）**

- Notion 公开文档**没有**专门说"账号被封后能否取回数据"——这是事实【M; Notion Help Center 多页查阅】
- "GDPR 不覆盖用户创作内容"**不准确**——GDPR Art. 15 (right of access) 在欧洲多次解读中**确实**覆盖用户上传内容；但执行层面被封号用户能否实际行使是另一回事【M; 法律解读不一致】
- LowEndBox 报道是 W 级单一事件【W; 调研已诚实标 W 级】
- Notion 帮助中心明确说"workspace 删除后 30 天内可联系 support 恢复"【M; Notion docs】
- **更精确表述应为**"Notion 不公开承诺账户被封后用户能强制取回数据；GDPR 实际可执行性在被封情况下未在公开判例中验证"——比"GDPR 不覆盖"弱一些但更准确

**对调研的修正建议**：把这条从"事实陈述"降级为"工程层面应假设的最坏情况"——这两者在工程决策上指向同一结论（不依赖 SaaS），但表述精确度不同。

### Fact 4：Airtable / Trello CSV 导出不包含附件 / 评论 / 活动历史

**调研引用**：notes/08 §三.3 "Airtable / Trello CSV 导出不包含附件、评论、活动历史、公式、关联记录"

**Fact-check 结果**：✅ **基本准确**

- Airtable CSV 不能保存 attachments（CSV 是 plain text，不能含二进制）；可用 formula field 提取 URL 但 URL 在外部 SaaS【M; Airtable Support / SwitchLabs guide】
- 复杂字段（formula、attachment、linked record）**不能正确表达**于 flat CSV【M; Airtable Community】
- 评论 / 活动历史的 CSV 导出能力**不存在原生支持**；需要 Enterprise eDiscovery API 或自建 API 抓取【M; Airtable Community 2024】
- 关联记录（linked record）CSV 导出会变成纯字符串 ID，不能 round-trip【M】

**Trello 我未单独检索 fact-check** ❓——但 Trello 也是 SaaS + JSON/CSV 导出格式，结论模式应类似。调研此点对 Airtable 准确，对 Trello 标 ❓ 更安全。

### Fact 5：Perez De Rosso & Jackson (OOPSLA 2016 / MIT Thesis 2017) git 复杂度研究

**调研引用**：notes/08 §2.2.1 "Perez De Rosso, S., & Jackson, D. (OOPSLA 2016 / MIT thesis 2017) 'Purposes, Concepts, Misfits, and a Redesign of Git'"

**Fact-check 结果**：✅ **准确**

- 论文确实于 OOPLSLA 2016 (SPLASH 2016 - OOPSLA track) 发表【S; 2016.splashcon.org/details/splash-2016-oopsla/34】
- 作者 Santiago Perez De Rosso & Daniel Jackson，MIT CSAIL【S; spderosso.github.io】
- 论文使用"概念设计分析"方法，并构造了 Gitless 替代设计【S; 论文摘要】
- 包含 Stack Overflow 问题分析 + 用户研究比较 git 与 Gitless【S】
- MIT thesis 版本（2017，dspace.mit.edu/handle/1721.1/106271）也存在【S】

**调研引用此点完全准确**——这是调研里学术引用最可靠的一项之一。

## 附：本 review 的最关键三条工程现实

按用户最终输出格式要求总结（从意见 1-30 中选最重要 3 条）：

### 1. **早晨规则书写是 v0 应该把所有工程预算压上去的最高 ROI 单一组件**【意见 25 + 9.3，等级 M】

理由：implementation intention 文献 + Patterson 2020 commitment device RCT + Masicampo 2011 open loop——三项 S 级证据汇合到同一方向。工程量 0-5 工时。"按时刻分层"虽然方向对，但 4 层并列处理模糊了这个 ROI 差异。**v0 = 早晨规则书写 + Obsidian + 周复盘 + 完全无 AI**——其他都是 v1+ 的渐增。

### 2. **"markdown + git GUI 封装"是被低估了至少 1 个数量级的工程承诺**【意见 1-2 + 1.4，等级 M】

理由：Obsidian + Obsidian Git plugin 是目前最接近"已封装好"的开源组合，但 plugin 是社区单人维护（denolehov），5 年维护风险不可忽略；移动端 fallback 弱；与 Obsidian Sync 互斥语义复杂。**git 在"防止用户骗自己"的工程强度比调研暗示弱**——必须 GPG 签名 + 推送 + 保护分支等 6 步配置全做对，对小白用户都不会做对。**调研在 v2 修订时应明确："git 审计"是理论上限，不是默认形态**。

### 3. **σ v0 应当明确放弃移动端介入，且明确"完全无 AI"**【意见 13 + 17 + 9.1，等级 M-S】

理由：iOS 沙盒 ✅ fact-check 通过——硬技术约束，不是产品偏好。Android 国产 ROM 浮窗权限复杂，对小白部署成本巨大。AI 集成默认走云 LLM API 与"本地优先 / 中等被审视感"原则**直接矛盾**——要做合理的 AI 集成需要 Ollama + 本地 7-8B 模型 + 32GB RAM 硬件门槛。**v0 同时撇清这两条，工程量减半，留存风险减半，临床安全自动满足**——这是同时取得"工程现实 + 临床证据 + Phase 1 v5 §三.11 兼容"的三赢组合。
