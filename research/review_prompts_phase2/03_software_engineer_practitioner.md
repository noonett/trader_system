具体评估任务：

## 1. 介质选型的工程现实核查

针对 markdown + git 作为推荐介质，请回答：
- ✅ 准确：git GUI 封装能否真正屏蔽小白门槛？哪些 GUI 在你实战中可用？哪些是 paper-tiger（看起来好用实际坑）？
- ⚠️ 部分准确：git 的"事后改写昂贵"在 force push / 本地回滚 / 删除 .git 等情况下被多大程度绕过？
- ❌ 错误：是否存在比 git 更适合个人单用户的"事后改写昂贵"机制？
- ❓ 不熟悉

针对 SQLite vs markdown + git 的对比：
- 应用层显式审计的 SQLite 是否真的"审计能力 ≪ git"？
- 在"AI 可分析性"维度上，SQLite 是不是显著优于 markdown + git？
- 文档说 "git GUI 封装是必须的" 是否有特定 GUI 推荐？这些 GUI 的真实用户留存如何？

## 2. SaaS 工具的 vendor lock-in 风险评估

文档警告：
- Evernote 翻倍 + 砍免费版（"笔记被人质化"）
- Notion 账户被封 = 数据无法找回（GDPR 不覆盖"用户创作内容"）
- Airtable / Trello CSV 导出**不包含**附件/评论/活动历史

请评估：
- 这些案例是否准确？
- "Notion GDPR 不覆盖用户创作内容"的法律事实是否准确？
- 是否有 SaaS 在数据可移植性上做得更好的反例（如 Obsidian Sync / Tana / Logseq Pro）？
- 你会推荐哪种数据策略？为什么？

## 3. 系统级浮窗（Raycast / Spotlight / Alfred）作为入口的工程实现

调研标记"系统级浮窗 = 高潜力 + 最弱证据"。请评估：
- Raycast / Spotlight / Alfred 作为"行为日志入口"的工程实现可行性
- iOS 沙盒禁止系统级介入是否真的是硬约束？
- 是否有跨平台（macOS / Windows / Linux）的浮窗实现方案？
- 浮窗 + Web 周报双形态的工程复杂度如何？

## 4. 中国 A 股客户端的物理可达性

调研指出：A 股 PC 客户端（同花顺/通达信，~21M MAU）是 Windows native，**浏览器扩展完全不可达**。

请评估：
- 这个事实是否准确？是否有第三方 SDK 或自动化方案？
- 在 Windows native 客户端旁边放"浮窗 / 提醒"的最简实现是什么？AutoHotkey？UiPath？Tasker？
- 移动端券商 App 的拦截可行性如何（iOS/Android 沙盒约束）？

## 5. "三个位置组合"的工程总成本

调研建议三个位置：
- **(a)** 早晨规则书写
- **(b)** 盘中拦截（仅当违反今早规则时）
- **(c)** 盘后 30 秒 EMA

工程上：
- 这三个位置如果在桌面 Web App 里实现，需要哪些核心组件？工作量？
- 如果在 Cursor IDE 里实现，是否合理？哪些功能 Cursor 天然不擅长？
- 如果选择"系统级浮窗 + Web 周报"组合，工程复杂度比单一桌面 Web App 高多少？

## 6. AI 集成的工程边界

调研建议 AI 是"按时刻分配的可选层"，包括"前台/后台/无 AI"三种形态。

工程上：
- 同一个系统支持"用户可切换 AI 形态"的代码复杂度如何？
- 后台 AI 静默分析的数据流（用户记录 → 周期性 AI 分析 → 异步推送）需要哪些组件？
- 完全 No-AI 模式下的"自动 lint 脚本（规则验证 / 红线触发器）"在工程上是什么？markdown lint？YAML schema 校验？自定义 Python 脚本？

## 7. PGSI / PHQ-4 主动筛查的工程实现

v5 §三.11 + Phase 2 调研要求 onboard 阶段做 PGSI + PHQ-4 筛查。

工程上：

- 量表实现是简单表单还是有 adaptive testing 需求？
- 阈值判定后的"系统拒绝接入主动训练 + 仅提供被动基线方案 + 转介信息"在产品形态上的最小可行实现是什么？

## 8. "git commit 仪式 + 私有 + AI 周期性透明读"的工程现实

调研建议"中等被审视感"的可行结构是：markdown + git commit 仪式 + 私有 + AI 周期性透明读。

工程上：

- 用户能多大程度上"绕过" AI 读取（删除文件 / 选择性提交 / 关闭 AI 集成）？
