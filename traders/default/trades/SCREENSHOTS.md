# 截图约定 — 交易记录的客观证据层

> **设计依据**：Keiser & Arthur 2021 显示客观媒介是 AAR 显著调节因子；Rojas 2020 / Mohamed 2020 / Sawyer 2012 一致表明视频/截图证据显著优于纯文字回忆。`foundation_2026.md` 约束："必须有客观证据字段（K 线截图 / 订单数据 / 时间戳）"。

---

## 目的

保留**下单时和结单时的决策环境**，用于日后复盘。截图解决的核心问题：

- 消除"事后回忆偏差"——你看到的市场和你记忆中的市场不一样
- 为周复盘提供客观锚点——AI 和你都能"回到当时的画面"
- 让"事前预设 → 实际发生 → 偏差"三段式对照有视觉基础

---

## 工作流（AI 代办）

**你只需要做两件事：**

1. **截图** — 下单/平仓时用手机或电脑截一张当时的画面
2. **贴给 AI** — 对话中发图 + 说一句话（如"这是今天 MGC 入场"）

**AI 负责所有后续：**
- 保存到正确路径（`trades/YYYY/MM/screenshots/`）
- 按约定命名
- 更新交易记录文件的截图引用
- git commit

**你不需要**：重命名文件、记住命名规则、手动填写文件名、手动放进目录。

---

## 什么时候截

| 时机 | 截什么 | 必要性 |
|------|--------|--------|
| **下单时** | 你做决策时看的那个 K 线画面（能看到入场位、止损位、当前价格） | 核心 |
| **平仓时** | 平仓时的 K 线画面（能看到实际出场位） | 核心 |
| **持仓中** | 你觉得重要的任何画面（盘口异动、突破/假突破、新闻） | 可选 |

**最低标准**：入场一张 + 出场一张 = 每笔交易 2 张。

**不需要**：
- 截图多完美——需要的是"真实"不是"漂亮"
- 每根 K 线都截
- 后期修图标注（除非关键价位不明显）

## 目录结构（AI 维护）

```
trades/YYYY/MM/
├── YYYY-MM-DD-<symbol>-<dir>.md        ← 交易记录
└── screenshots/
    ├── YYYY-MM-DD-<symbol>-<dir>-entry-1.png
    ├── YYYY-MM-DD-<symbol>-<dir>-exit-1.png
    └── YYYY-MM-DD-<symbol>-<dir>-context-1.png  (可选)
```

### 命名规则（AI 执行，你不用记）

```
{trade-id}-{type}-{seq}.{ext}
```

- `trade-id` = 与交易 `.md` 文件名相同
- `type` = `entry` | `exit` | `context`
- `seq` = 序号（同类多张时 1, 2, 3...）

## Git 管理

截图是二进制文件。当前阶段（v0，交易频率低），直接 `git add` 即可。一张 PNG 截图约 100KB-500KB，每月 10-20 笔交易 = 每月约 2-10MB，Git 完全承受得住。

如果仓库体积增长明显，后续可迁移到 Git LFS：

```bash
git lfs track "trades/**/screenshots/*.png"
git lfs track "trades/**/screenshots/*.jpg"
git lfs track "trades/**/screenshots/*.webp"
```

## 复盘时的价值

### 周复盘

AI 周报引用截图做对照分析：
- "入场时你看到了什么（entry 截图） → 实际走势（exit 截图） → 偏差在哪"

### 月度/季度回顾

连续多笔的 entry 截图放在一起看 = **决策环境时间线**，能发现：
- "我是不是连续在同一个位置入场"（模式识别）
- "连续亏损后我的入场位是不是变激进了"（行为偏误可视化）
